# S1.5 — Deterministic CSS attribution for contrast

> **Goal:** stop sending whole stylesheets to the LLM. For `color-contrast` (and other
> CSS-governed) violations, find the exact CSS rule(s) that style the offending element
> **in code** (parse + selector-match + specificity), and hand the model only those few
> rules. Then map the winning **compiled** rule back to its **SCSS source** so the fix lands
> in the editable file, not the build artifact.
> **Plan:** [§3 Fix scope / CSS](../a11y-agent-plan.md), [§5 loop TRIAGE+READ+FIX](../a11y-agent-plan.md), [§9 CSS attribution](../a11y-agent-plan.md).
> **Spike:** validated — see `core-web/scratch/SPIKE-css-attribution.md` + `css-attribution-proto.ts`.

## Why this session exists

S1's loop sent every candidate file (incl. full CSS) to the LLM on **every** triage call.
On the demo theme that's ~34K tokens of compiled CSS per call — **~$1–12 per page** and it
**doesn't scale** (a customer with many SCSS files would blow the context window). CSS
contrast is the **most common** violation class, so this is the difference between the agent
being usable and not.

The spike proved the fix: deterministic attribution cuts the CSS payload **34,007 → ~106
tokens (99.7%)** and is **scale-independent** (1 stylesheet or 140 partials, same cost).

## Entry state
- S1 done: headless `POST /fix` → §6 report; the loop reads candidate files and runs
  triage/fix via the LLM.
- `_render-sources` returns `theme.files[]` with `{ extension, identifier, path }` —
  **all** theme files (vtl, css, scss, js, images, fonts). `runFix.collectCandidates`
  already filters to `EDITABLE_THEME_EXTENSIONS` (vtl/css/scss/sass/dotsass/less; JS excluded).
- Spike deps added `-D`: `css-select`, `htmlparser2`, `domhandler` (+ `postcss`, already in tree).

## Tasks (in order)
1. **`css-attribution` module** (pure functions over strings — zero API cost to test):
   - `parseColorRules(css)` → rules `{ selector, decls[], sourceLine }`, keeping only
     color-affecting decls (`color`, `background`, `background-color`, `border-color`, `fill`).
   - `attribute(elementHtml, rules)` → the matching rules, ranked by specificity:
     - build a DOM node from the violation's `context` HTML (`htmlparser2`),
     - match with `css-select` — **SOUND matching only: pure compound selectors** (no
       descendant/child/sibling combinator), since the scan gives the element but **not its
       ancestors**. Strip dynamic pseudos (`:hover/:focus/::before`) before matching, keep in
       output. (The rightmost-compound fallback is **unsound** — it false-positives; the spike
       proved this. Excluding combinator selectors trades recall for precision.)
   - Promote `@csstools/selector-specificity` over the spike's hand-rolled ranker (optional).
2. **Wire into triage** (`triage.ts`): for CSS-fixable findings, run `attribute(...)` over
   the read CSS and pass **only the matched rules** to the LLM, not whole files. VTL findings
   keep their current path. This is where the token win lands.
3. **Compiled → SCSS source mapping** (the other half — so fixes persist):
   - The page renders **compiled** CSS (`styles_precompiled.css`), but the editable source is
     **SCSS** (`styles.scss` + the partials under `css/custom-styles/**`, `css/plugins/**`).
     Editing compiled CSS is **futile** — regenerated on next compile.
   - Locate the winning selector/value in the SCSS tree (search the matched selector across
     `.scss` candidates; account for SCSS **variables** — the literal hex may live in
     `_variables-custom.scss`, not next to the selector — and mixins). Prefer a sourcemap if
     the compile emits one.
   - The FIX edits (and saves) the **SCSS** file. If the source can't be located confidently
     → `reported` with guidance (don't edit the artifact).
4. **Contrast math in code (optional but recommended):** compute the ratio (`wcag-contrast`
   / `culori`) and the minimal nudge to clear AA (4.5:1 normal, 3:1 large), leaving the LLM
   only genuinely ambiguous cases. Reduces LLM calls *and* makes the nudge deterministic.
5. **Promote spike deps to real deps** (move `css-select`/`htmlparser2`/`domhandler` out of
   devDependencies if the module ships them) and add unit tests (real demo CSS fixtures + the
   `#book` / `.button-primary` case from the spike).

## Contract to honor
- **No CSS bytes to the LLM** beyond the few attributed rules. The whole point.
- The §6 report is unchanged. A CSS fix still reports `file` = the **SCSS** path,
  `blastRadius` (`element-scoped | shared-rule | token`), and `review`.
- Attribution runs **in the agent** (deterministic); it is not an LLM tool call.

## Scope fences (do NOT)
- Don't send whole stylesheets to the model anymore (that's the regression this prevents).
- Don't edit compiled/precompiled CSS or minified assets — SCSS source only; else report.
- Don't try to recover descendant-selector recall by matching rightmost compounds (unsound).
  Recovering it properly needs **ancestor context from the scanner** — a separate enhancement,
  not this session.
- No proxy/Studio/streaming work.

## Definition of done
- A `color-contrast` violation on a real page is attributed to the correct rule
  (`.button-primary { background-color:#e76300 }` for the demo "Book now" button) and the LLM
  receives only the matched rules — verified by the usage sink showing the per-triage CSS
  token count dropped from tens-of-thousands to hundreds.
- The applied fix lands in the **SCSS source** file (or reports cleanly when the source can't
  be located), and the working re-scan confirms the violation cleared.
- `css-attribution` has unit tests (pure, no API) incl. the sound-matching guard (a
  descendant selector like `.x .y a` must NOT match a lone `<a>`).
