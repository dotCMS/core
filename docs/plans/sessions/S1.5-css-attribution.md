# S1.5 — Deterministic CSS attribution for contrast

> **Goal:** stop sending whole stylesheets to the LLM. For `color-contrast` (and other
> CSS-governed) violations, find the exact CSS rule(s) that style the offending element
> **in code** (parse + selector-match + specificity), and hand the model only those few
> rules. Then map the winning **compiled** rule back to its **SCSS source** so the fix lands
> in the editable file, not the build artifact.
> **Plan:** [§3 Fix scope / CSS](../a11y-agent-plan.md), [§5 loop TRIAGE+READ+FIX](../a11y-agent-plan.md), [§9 CSS attribution](../a11y-agent-plan.md).
> **Spike:** validated — see `core-web/scratch/SPIKE-css-attribution.md` + `css-attribution-proto.ts`.

## Why this session exists

S1's loop sent every candidate file (incl. full CSS) to the LLM on **every** triage call.
On the demo theme that's ~34K tokens of compiled CSS per call — **~$1–12 per page** and it
**doesn't scale** (a customer with many SCSS files would blow the context window). CSS
contrast is the **most common** violation class, so this is the difference between the agent
being usable and not.

The spike proved the fix: deterministic attribution cuts the CSS payload **34,007 → ~106
tokens (99.7%)** and is **scale-independent** (1 stylesheet or 140 partials, same cost).

## Core principle — NEVER pre-read the theme tree (locked)

The real problem isn't context-window size — it's that **handing 150 files to any model is the
wrong operation.** Real themes have 150+ SCSS partials (the demo `travel` theme: **154
editable files ≈ 217K tokens**); a customer could have far more. Sending them all is expensive,
*less* accurate (the relevant rule is buried in bootstrap noise), slow, and doesn't generalize.
No model fixes this — the architecture must.

**The loop must be LAZY and per-violation. It must NOT read the file tree up front.** Per
violation, touch exactly:

```
violation (element + selector)
  → attribute against the ONE compiled stylesheet the page loads   (postcss + css-select, no LLM)
  → winning rule (~100 tokens)                                     (the only thing the LLM sees)
  → sourcemap (column-level) → the ONE source file + line          (your endpoint)
  → READ that one file → edit → save
```

So the partial count is **irrelevant** — 5 or 5,000, a fix reads ~1 compiled stylesheet + ~1
source file. **Abandoned approach:** `collectCandidates` pre-reading every editable theme file
into `currentContent` and passing them to triage. That is exactly what blows up (217K tokens →
"Provider returned error" on Kimi; ~$1–12/page on Anthropic). Rip it out.

## Entry state
- S1 done: headless `POST /fix` → §6 report. **Current loop pre-reads the whole editable
  theme tree** (`collectCandidates` + `currentContent`) and passes it to triage — this is the
  thing to remove (see Core principle).
- `_render-sources` returns `theme.files[]` with `{ extension, identifier, path }` — **all**
  theme files (154 editable on the demo theme). Used now to find the **compiled stylesheet**
  and to resolve a sourcemap `sources[]` path back to an asset, NOT to bulk-read.
- **SASS sourcemap endpoint shipped & validated** (`?dotsass=true&sourcemap=true`): inline,
  column-level, `sourcesContent` embedded, `sources[]` theme-relative. Maps a compiled color
  value → its `$variable` source file+line. See `scratch/SPEC-sass-sourcemap-endpoint.md`.
- Deps: `postcss` (in tree), `css-select`/`htmlparser2`/`domhandler` (-D, spike), `source-map`
  0.7.6 (-D, sourcemap decode). Provider default is OpenRouter + `moonshotai/kimi-k2.7-code`.

## Tasks (in order)
1. **Rip out the pre-read.** Remove `collectCandidates` bulk-reading every editable theme
   file into `currentContent`. The loop must not read the tree. VTL keeps a *small* candidate
   set (theme VTLs + container VTLs from `_render-sources` — ~10 refs); CSS goes through the
   lazy path below.
2. **`css-attribution` module** (pure functions over strings — zero API cost to test):
   - `parseColorRules(css)` → rules `{ selector, decls[], pos }`, keeping only color-affecting
     decls (`color`, `background`, `background-color`, `border-color`, `fill`).
   - `attribute(elementHtml, rules)` → matching rules, ranked by specificity:
     - build a DOM node from the violation's `context` HTML (`htmlparser2`),
     - match with `css-select` — **SOUND matching only: pure compound selectors** (no
       descendant/child/sibling combinator), since the scan gives the element but **not its
       ancestors**. Strip dynamic pseudos (`:hover/:focus/::before`) before matching, keep in
       output. (The rightmost-compound fallback is **unsound** — false-positives; spike proved it.)
   - Keep the matched **postcss node** (for surgical edit + its position, for the sourcemap lookup).
3. **The lazy CSS path** (per violation, in the loop):
   - Fetch the **one compiled stylesheet** the page loads (`styles.scss?dotsass=true&sourcemap=true`
     — also gives the sourcemap in the same response). Parse + `attribute(...)` → winning rule.
   - The LLM sees **only the matched rule** (~100 tokens) — or skip the LLM entirely with
     contrast math (task 5).
4. **Compiled → SCSS source via the sourcemap** (shipped & validated):
   - Decode the inline map (`data:application/json;charset=utf-8,…` → `decodeURIComponent`, NOT
     base64) with `source-map`. For the matched decl's **value column**, `originalPositionFor`
     → the source `.scss` file + line (lands on the `$variable` when the value is a variable).
   - `sources[]` are theme-relative → resolve to the asset path (via `_render-sources` /
     theme folderPath) and READ **only that one file** to edit. `sourcesContent` is embedded,
     so the source text is in the map too. Edit the **SCSS**; never the compiled artifact. If
     the position can't be resolved → `reported`.
5. **Contrast math in code (recommended):** compute ratio + minimal nudge to clear AA (4.5:1
   normal, 3:1 large) with `wcag-contrast`/`culori`, so the color change is deterministic and
   the LLM is used only for genuinely ambiguous cases (or not at all for contrast).
6. **Promote deps + tests:** move `css-select`/`htmlparser2`/`domhandler`/`source-map` to real
   deps; unit-test attribution (incl. the sound-matching guard) + sourcemap resolution against
   the demo fixtures (`#book`/`.button-primary` → `_variables-custom.scss`).

## Contract to honor
- **No CSS bytes to the LLM** beyond the few attributed rules. The whole point.
- The §6 report is unchanged. A CSS fix still reports `file` = the **SCSS** path,
  `blastRadius` (`element-scoped | shared-rule | token`), and `review`.
- Attribution runs **in the agent** (deterministic); it is not an LLM tool call.

## Scope fences (do NOT)
- Don't send whole stylesheets to the model anymore (that's the regression this prevents).
- Don't edit compiled/precompiled CSS or minified assets — SCSS source only; else report.
- Don't try to recover descendant-selector recall by matching rightmost compounds (unsound).
  Recovering it properly needs **ancestor context from the scanner** — a separate enhancement,
  not this session.
- No proxy/Studio/streaming work.

## Definition of done
- A `color-contrast` violation on a real page is attributed to the correct rule
  (`.button-primary { background-color:#e76300 }` for the demo "Book now" button) and the LLM
  receives only the matched rules — verified by the usage sink showing the per-triage CSS
  token count dropped from tens-of-thousands to hundreds.
- The applied fix lands in the **SCSS source** file (or reports cleanly when the source can't
  be located), and the working re-scan confirms the violation cleared.
- `css-attribution` has unit tests (pure, no API) incl. the sound-matching guard (a
  descendant selector like `.x .y a` must NOT match a lone `<a>`).
