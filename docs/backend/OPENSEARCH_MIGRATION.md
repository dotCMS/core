# Test Plan — OpenSearch Migration: Startup Hardening, haltMigration(), Phase-Aware Index Init

**Mode: qa-prefix (PR #35632)**

These TCs define the acceptance criteria for PR #35632 and the migration scenarios not yet covered
by the existing plan in `docs/backend/OPENSEARCH_MIGRATION_TEST_PLAN.md`. All cases are **Manual**
because they require Docker, a live dotCMS instance, and external tools (Kibana, OS Dashboards).
Source of truth: `docs/backend/OPENSEARCH_MIGRATION.md`.

## Summary

PR #35632 changes the startup behavior for Phase ≥ 1 when OS is unreachable or reports the wrong
version: instead of continuing (previous behavior), it now calls `haltMigration()` and resets the
phase to 0. Additionally, `hasEmptyIndices()` is now phase-aware (prevents spurious full reindexes)
and `threadSafeTimestampFormatter` replaces `SimpleDateFormat` under concurrent load.

This plan covers: (1) the 5 new PR behaviors, (2) invalid endpoint values, (3) multi-node cluster,
(4) Content Type and contentlet lifecycle sync, (5) index delete variants, (6) complex query
comparison between Kibana and OpenSearch Dashboards.

The existing plan (`docs/backend/OPENSEARCH_MIGRATION_TEST_PLAN.md` — Scenarios A–F) remains valid
for base scenarios. **NOTE:** Scenario B-7 of that plan is incorrect after PR #35632 — see TC-001
(Replaces: B-7).

### dotCMS Configuration

All properties below go in `dotmarketing-config.properties` (or `dotcms-config-cluster.properties`).

#### Minimal OS connection (no TLS, no security)

```properties
OS_ENDPOINTS=http://localhost:9201
OS_AUTH_TYPE=BASIC
OS_AUTH_BASIC_USER=admin
OS_AUTH_BASIC_PASSWORD=admin
OS_TLS_ENABLED=false
```

#### Migration phase flag

```properties
# 0 = ES only (default)
# 1 = dual-write, ES reads
# 2 = dual-write, OS reads  (read layer not yet migrated — partial testing only)
# 3 = OS only              (NOT TESTABLE — do not activate)
FEATURE_FLAG_OPEN_SEARCH_PHASE=0
```

> The flag is read at startup **and** on each routing decision. A restart is not required for
> the change to take effect, but all nodes in a cluster must be updated consistently.

#### Confirming the active phase at runtime

```bash
# Check the startup log for a line like:
#   Migration Phase: PHASE_1_DUAL_WRITE_ES_READS
grep -i "migration phase" <dotcms-log-file>

# Or query the index list and inspect which backends respond:
curl -s -u admin:admin http://localhost:8082/api/v1/esindex | jq .
```

## Migration Phase Reference

| Phase | Code Name                     | Writes   | Reads    | ES Status            | OS Write Failures        |
|-------|-------------------------------|----------|----------|----------------------|--------------------------|
| 0     | PHASE_0_MIGRATION_NOT_STARTED | ES only  | ES only  | Active               | n/a                      |
| 1     | PHASE_1_DUAL_WRITE_ES_READS   | ES + OS  | ES only  | Active               | Fire-and-forget (logged) |
| 2     | PHASE_2_DUAL_WRITE_OS_READS   | ES + OS  | OS only  | Active (fallback)    | Fire-and-forget (logged) |
| 3     | PHASE_3_OPENSEARCH_ONLY       | OS only  | OS only  | Decommissioned       | Propagate to caller      |

**Testable scope for this plan:**

| Phase | Testable? | Limitation |
|-------|-----------|------------|
| 0     | Yes — fully | No OS dependency |
| 1     | Yes — fully | Dual-write verified via Kibana + OS Dashboards |
| 2     | Partially | Dual-write testable via OS Dashboards; application reads still resolve against ES |
| 3     | **No** | Query layer not migrated — activating Phase 3 produces broken application reads |

### `os::` tag rule (important for DB checks)

The `os::` prefix is a **DB-only artifact** used to prevent name collisions in the shared `indicies`
table. It is added by `VersionedIndicesAPIImpl` on write and stripped on every read. Callers and
REST responses always receive the clean logical name — the prefix is never visible outside the DB.

```sql
-- OS index rows in DB look like:  os::cluster_<id>.working_20260421120000
-- REST API and client tools see:      cluster_<id>.working_20260421120000
```

## DB Validation Queries

Run these against the dotCMS PostgreSQL database at any point during test execution to verify
index state:

```sql
-- All index rows (both ES and OS)
SELECT * FROM indicies ORDER BY index_name;

-- OS index rows only
SELECT * FROM indicies WHERE index_name LIKE 'os::%';

-- ES index rows only
SELECT * FROM indicies WHERE index_name NOT LIKE 'os::%';

-- Verify a specific index was removed after delete
SELECT * FROM indicies WHERE index_name = '<deleted_index_name>';
SELECT * FROM indicies WHERE index_name = 'os::<deleted_index_name>';

-- Count rows per provider
SELECT
  CASE WHEN index_name LIKE 'os::%' THEN 'OS' ELSE 'ES' END AS provider,
  COUNT(*) AS row_count
FROM indicies
GROUP BY 1;

-- Expected state in Phase 1 (fresh install): 4 rows — 2 ES + 2 OS
-- Expected state in Phase 0: 2 rows — 2 ES only (no os:: rows)
```
